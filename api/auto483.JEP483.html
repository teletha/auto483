JEP483

Provides support for JEP 483 (Ahead-of-Time Class Loading Linking).

This utility class helps in managing the creation and usage of Ahead-of-Time (AOT) cache files introduced in Java 24. It aims to simplify the process of leveraging AOT compilation to potentially improve application startup performance.

The #enable() methods check for the existence of an AOT cache file. If the file doesn't exist, this class orchestrates the necessary steps to generate it. This might involve restarting the application in a special "record" mode or setting up a shutdown hook to create the cache upon exit.

JEP483()

enable()void

Enables Ahead-of-Time (AOT) loading and linking using the default cache file name .aot.

This is a convenience method that calls #enable(String) with ".aot" as the cache file name.

enable(StringcacheFileName)void

StringcacheFileName

The name (or path) of the AOT cache file to use or create. This name is used for both the AOT cache (-XX:AOTCache) and the AOT configuration file (-XX:AOTConfiguration, with &quot;conf&quot; appended).

Enables Ahead-of-Time (AOT) loading and linking using the specified cache file name.

This method checks if the specified AOT cache file exists. If it does not exist, it manages the process of generating the cache based on the current JVM execution mode:

  • Normal Mode (Attempting to use cache): If the JVM was started with -XX:AOTCache=<cacheFileName> but the file is missing, this method will restart the current application with the -XX:AOTMode=record flag. The current process will be terminated (System.exit(0)). The restarted process will then run the application and record AOT compilation data.
  • Record Mode: If the JVM was started with -XX:AOTMode=record, this method registers a shutdown hook. When the application exits normally, the shutdown hook will launch a new JVM process with -XX:AOTMode=create to compile the recorded data into the final AOT cache file.

If the cache file already exists, this method does nothing, allowing the JVM to load the AOT cache as configured.

Note: The cache generation process involves starting new JVM processes. Ensure that the necessary permissions and environment setup are in place for this to succeed. The classpath used for the cache generation process is derived from the current application's classpath.
